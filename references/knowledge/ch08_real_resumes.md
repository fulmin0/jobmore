# Ch8: Real Resumes

## Summary
Annotated examples of actual PM resumes — good and bad — with commentary on what works and what doesn't. Shows the 5 rules from Ch7 applied to real documents. Also covers résumé examples by background type: engineer-to-PM, MBA-to-PM, new grad, experienced PM. High value for seeing the rules in action.

## Key Concepts
- **Before/after patterns** (common in annotated examples):
  - Responsibility-style bullets → accomplishment-style rewrites
  - Vague outcome ("improved experience") → quantified outcome ("reduced load time by 40%")
  - Paragraph blocks → bullet lists
  - Buried lead → restructured to lead with the most impactful bullet
- **Engineer-to-PM resume rules**:
  - Lead with products shipped, not technologies used
  - Translate "built the API" → "launched the API that enabled X integrations, reducing onboarding time by Y%"
  - Avoid listing languages/frameworks as primary bullets — put in Skills section
- **MBA-to-PM resume rules**:
  - Show execution, not just strategy or analysis
  - "Developed a framework" is weak without a shipped outcome
  - Internship / APM experience should lead if it's the strongest PM signal
- **New grad resume rules**:
  - Projects > coursework > GPA
  - A shipped side project outweighs any class project
  - Internship bullets must be outcome-focused ("shipped X to N users"), not task-focused ("worked on Y feature")
- **Experienced PM resume rules**:
  - Each role needs 2–4 bullets max — force-rank to the most impactful
  - Scale and scope must be explicit: "team of N engineers", "M MAU", "$X ARR product"
  - Summary line is most valuable here — positions the candidate before the reader hits the work history
- **Red flags reviewers notice immediately**:
  - No metrics anywhere in a 5+ year resume
  - Same bullet style for every role (shows lack of thought)
  - Dates missing or inconsistent
  - Skills section that lists every tool ever touched (signals no prioritization ability)

## Direct Quotes / Signals Worth Preserving
- Annotated example commentary: "This bullet tells me what you did. It doesn't tell me why it mattered. Rewrite to show the outcome."
- On the skills section: "A PM who lists 40 tools has told me nothing. A PM who lists 6 and can speak to each one — that's a skills section."
- On engineer-to-PM transition: "Your technical background is the credential. Your product thinking is the pitch."

## Relevance to Job Search Pipeline
- **Resume content audit**: Before finalizing any set of bullets, run the "before/after" test from this chapter — does each bullet describe an outcome, or just an action?
- **Role-specific framing**: For cross-industry applications, use the engineer-to-PM translation pattern — lead with product shipped, not technology used
- **Red flag checklist**: Add to the pre-submission scan in `feedback_resume_scan.md`: No metrics, same bullet style, missing dates, bloated skills section
- **Experience file enrichment**: When enriching `data/experience/` files, identify which existing bullets are "responsibility" style and flag for rewrite — use Ch7+Ch8 framing
